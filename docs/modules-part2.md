## 模块 9：MCP 集成

### 这是什么？

MCP（Model Context Protocol）是一个开放协议，让 Claude Code 能连接外部工具和服务。通过 MCP，Claude 不仅能操作本地文件，还能操作数据库、调用云服务 API、管理 Kubernetes 集群……

### 用一个比喻来理解

如果 Claude Code 本身自带的工具是"瑞士军刀"（Bash、FileRead 等），那 MCP 就是一个**工具接口标准**，可以让你插上无限种"电动工具"：

```
  瑞士军刀 (内置工具)          MCP 接口 (外部工具)
  ┌────────────────┐          ┌────────────────┐
  │ Bash           │          │ Azure MCP      │  ← 管理 Azure 资源
  │ FileRead       │          │ GitHub MCP     │  ← 操作 GitHub API
  │ FileEdit       │          │ Database MCP   │  ← 查询数据库
  │ Grep           │          │ Kubernetes MCP │  ← 操作 K8s 集群
  │ WebSearch      │          │ Slack MCP      │  ← 发消息到 Slack
  │ ...            │          │ 你自建的 MCP   │  ← 任何你想要的
  └────────────────┘          └────────────────┘
         │                           │
         └───────── 都在 Claude Code 里统一使用 ───────┘
```

### MCP 怎么工作？

```
┌─────────────── Claude Code (客户端) ───────────────┐
│                                                     │
│  Claude AI: "我要查一下 Azure 上有哪些虚拟机"         │
│       │                                             │
│       ▼                                             │
│  MCPTool.execute({                                  │
│    server: "azure",                                 │
│    tool: "compute_list",                            │
│    arguments: { subscription: "xxx" }               │
│  })                                                 │
│       │                                             │
└───────┼─────────────────────────────────────────────┘
        │
        │  通过某种传输协议发送请求
        │
        ▼
┌─────────────── MCP 服务器 ────────────────┐
│                                           │
│  Azure MCP Server                         │
│                                           │
│  收到请求 → 调用 Azure API → 返回结果      │
│                                           │
│  结果: [                                   │
│    { name: "vm-prod-01", status: "Running" },│
│    { name: "vm-dev-02", status: "Stopped" }  │
│  ]                                         │
│                                           │
└───────────────────────────────────────────┘
```

### 传输协议——MCP 服务器怎么"连上"？

```
┌────────────────── 6 种连接方式 ──────────────────┐
│                                                  │
│  stdio (最常用)                                   │
│  ┌──────────────────────────────────────┐        │
│  │  Claude Code 启动一个子进程            │        │
│  │  通过 stdin/stdout 通信               │        │
│  │                                      │        │
│  │  就像: 你启动一个程序，               │        │
│  │  通过"对话框"给它发指令、收结果        │        │
│  │                                      │        │
│  │  配置: { command: "npx",             │        │
│  │          args: ["@azure/mcp"] }      │        │
│  └──────────────────────────────────────┘        │
│                                                  │
│  http / sse / ws (远程服务)                       │
│  ┌──────────────────────────────────────┐        │
│  │  通过网络连接到远程 MCP 服务器         │        │
│  │                                      │        │
│  │  就像：访问一个网站 API               │        │
│  │                                      │        │
│  │  配置: { url: "https://mcp.example"  │        │
│  │          type: "http" }              │        │
│  └──────────────────────────────────────┘        │
│                                                  │
│  sdk (进程内)                                     │
│  ┌──────────────────────────────────────┐        │
│  │  MCP 服务器直接在同一个进程里运行       │        │
│  │  最快，但不常用                        │        │
│  └──────────────────────────────────────┘        │
│                                                  │
│  sse-ide / ws-ide (IDE 桥接)                     │
│  ┌──────────────────────────────────────┐        │
│  │  通过 IDE 扩展的连接来通信             │        │
│  │  VS Code 扩展用这个                    │        │
│  └──────────────────────────────────────┘        │
│                                                  │
└──────────────────────────────────────────────────┘
```

### MCP 配置从哪里来？

```
  ┌──────── 多来源配置合并 ────────┐
  │                                │
  │  .mcp.json        (项目目录)   │ ← 团队共享的 MCP 配置
  │       │                        │
  │       ▼                        │
  │  ~/mcp.json       (用户目录)   │ ← 你个人常用的 MCP
  │       │                        │
  │       ▼                        │
  │  settings.json    (设置文件)   │ ← settings 里也能配
  │       │                        │
  │       ▼                        │
  │  插件带来的 MCP   (插件)       │ ← 插件自带的服务器
  │       │                        │
  │       ▼                        │
  │  ┌──────────────────────────┐ │
  │  │   合并 + 去重 + 权限过滤  │ │
  │  │                          │ │
  │  │   如果同一个服务器在多处   │ │
  │  │   配置了，只保留一个       │ │
  │  │                          │ │
  │  │   如果公司策略禁止某个     │ │
  │  │   服务器，自动过滤掉       │ │
  │  └──────────────────────────┘ │
  │                                │
  └────────────────────────────────┘
```

### MCP 服务器的 5 种状态

```
  ┌─────────┐     ┌─────────┐     ┌─────────┐
  │ Pending │ ──→ │Connected│     │ Failed  │
  │ 连接中   │     │ 已连接   │     │ 连接失败│
  └─────────┘     └─────────┘     └─────────┘

  ┌──────────┐     ┌──────────┐
  │NeedsAuth │     │ Disabled │
  │ 需要认证  │     │ 已禁用   │
  └──────────┘     └──────────┘

  每个 MCP 服务器连接后提供:
  • 工具列表 (可以调用哪些功能)
  • 资源列表 (可以访问哪些数据)
```

### 它和谁打交道？

```
  配置系统  ──→  MCP 配置 (哪些服务器、怎么连)
                    │
                    ▼
              MCPConnectionManager
              (管理所有 MCP 连接)
                    │
         ┌──────────┼──────────┐
         ▼          ▼          ▼
    MCP Client 1  Client 2  Client 3
         │          │          │
         ▼          ▼          ▼
    外部服务器    外部服务器   外部服务器
         │          │          │
         └──────────┼──────────┘
                    │
         工具注册到 tools.ts
         AI 可以像使用内置工具一样使用 MCP 工具
```

## 模块 10：插件系统 (Plugin System)

### 这是什么？

插件系统让 Claude Code 可以被社区扩展——任何人都可以编写插件，添加新命令、新工具、新技能甚至新的 MCP 服务器。

### 用一个比喻来理解

就像浏览器的扩展商店：
- **Chrome** 本身有基本功能（浏览网页）
- **Chrome 扩展** 给它加超能力（广告拦截、密码管理、翻译）

类似地：
- **Claude Code** 本身有基本功能（代码编辑、搜索）
- **插件** 给它加超能力（特定框架支持、CI/CD 集成、团队工具）

### 插件能提供什么？

```
┌──────────────── 一个插件可以包含 ──────────────────┐
│                                                    │
│  ┌──────────┐  新的斜杠命令                         │
│  │ Commands │  比如: /deploy, /lint, /format        │
│  └──────────┘                                      │
│                                                    │
│  ┌──────────┐  新的 AI 技能                         │
│  │ Skills   │  比如: "React 最佳实践检查器"          │
│  └──────────┘                                      │
│                                                    │
│  ┌──────────┐  新的工具                             │
│  │ Tools    │  比如: "Docker 容器管理工具"           │
│  └──────────┘                                      │
│                                                    │
│  ┌──────────┐  生命周期钩子                         │
│  │ Hooks    │  比如: "每次工具执行后记录日志"         │
│  └──────────┘                                      │
│                                                    │
│  ┌──────────┐  自带 MCP 服务器                      │
│  │ MCP      │  比如: "自带数据库连接器"              │
│  └──────────┘                                      │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 插件的生命周期

```
  ① 发现                    ② 安装                    ③ 加载
  ┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
  │ 扫描插件来源:     │      │ 从 Git 仓库克隆  │      │ 读取插件清单     │
  │                  │      │ 到本地缓存       │      │ 注册命令/工具    │
  │ • 市场 (远程仓库)│ ──→  │                  │ ──→  │                  │
  │ • 内置 (随 CLI)  │      │ 检查版本         │      │ 缓存加速         │
  │ • 临时 (--plugin)│      │ 安装依赖         │      │ (下次更快)       │
  └─────────────────┘      └─────────────────┘      └─────────────────┘

  ④ 运行                    ⑤ 刷新
  ┌─────────────────┐      ┌─────────────────┐
  │ 插件的命令/工具   │      │ 检测市场更新     │
  │ 像内置的一样工作  │      │ 热重载新版本     │
  │                  │      │                  │
  │ 用户无感知       │      │ /reload-plugins  │
  └─────────────────┘      └─────────────────┘
```

### 三种插件来源

```
  ┌─────────── Marketplace 插件 ────────────┐
  │  远程 Git 仓库里的插件                    │
  │  类似于 npm 包                            │
  │  通过 settings.json 的 plugins.enabled 启用│
  │  自动安装 + 更新                          │
  └──────────────────────────────────────────┘

  ┌─────────── Built-in 插件 ──────────────┐
  │  随 Claude Code CLI 一起发布             │
  │  不需要安装，开箱即用                     │
  │  核心功能扩展                             │
  └──────────────────────────────────────────┘

  ┌─────────── Inline 插件 ─────────────────┐
  │  通过 --plugin-dir 临时加载               │
  │  只在当前会话有效                          │
  │  适合开发和测试插件                        │
  └──────────────────────────────────────────┘
```

### 它和谁打交道？

```
  配置系统 ──→ "启用了哪些插件"
       │
       ▼
  PluginInstallationManager
       │
       ├──→ 命令系统 (注册新命令)
       ├──→ 工具系统 (注册新工具)
       ├──→ 技能系统 (注册新技能)
       ├──→ MCP 系统 (注册新 MCP 服务器)
       └──→ Hook 系统 (注册生命周期钩子)
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/services/plugins/pluginOperations.ts` | 插件安装/卸载/启用/禁用逻辑 |
| `src/services/plugins/pluginCliCommands.ts` | 插件相关 CLI 命令 |
| `src/services/plugins/PluginInstallationManager.ts` | 插件安装工作流管理 |
| `src/utils/plugins/pluginLoader.ts` | 插件加载器（带缓存） |
| `src/utils/plugins/marketplaceManager.ts` | 插件市场管理 |

## 模块 11：技能系统 (Skill System)

### 这是什么？

技能（Skill）是"预设的提示词模板"——把 Claude 要做的事情预先写好指令，用户一键触发。

### 用一个比喻来理解

技能就像**预设菜谱**：

- 不用技能：你对厨师说 "做个菜"，厨师不知道做什么
- 用了技能：你说 "/宫保鸡丁"，厨师拿到一份详细菜谱，按步骤做就行

在 Claude Code 里：
- 不用技能：你说 "帮我提交代码"，效果可能不稳定
- 用了技能：你说 `/commit`，Claude 拿到精心编写的 commit 指令，每次表现一致

### 技能和命令有什么区别？

```
  ┌─────────────────── 容易混淆的概念 ─────────────────────┐
  │                                                        │
  │  技能 (Skill)        命令 (Command)                     │
  │  ─────────────       ─────────────                     │
  │  一种提示词模板        用户的快捷操作                      │
  │                                                        │
  │  特点:                                                  │
  │  • AI 可以自动调用     • 只有用户手动输入 /xxx            │
  │  • 可以来自插件        • 可以是本地/智能/UI 任意类型       │
  │  • 核心是提示词        • 核心是执行逻辑                   │
  │                                                        │
  │  关系:                                                  │
  │  很多命令本质上是对技能的"用户友好包装"                     │
  │  比如 /commit 命令 = commit 技能 + 用户友好的触发方式      │
  │                                                        │
  └────────────────────────────────────────────────────────┘
```

### 技能的执行模式

```
  ┌──────── inline (内联) ─────────┐
  │                                │
  │  技能的提示词直接注入当前对话     │
  │  就像有人在对话框里帮你打了一段话 │
  │                                │
  │  适合: 简单任务、需要上下文的任务 │
  │  例子: /simplify (基于当前对话)  │
  └────────────────────────────────┘

  ┌──────── fork (派生) ──────────┐
  │                               │
  │  启动一个"子代理"来执行技能     │
  │  有自己独立的对话上下文          │
  │                               │
  │  适合: 复杂任务、不影响当前对话  │
  │  例子: /review (独立完成审查)   │
  └───────────────────────────────┘
```

### 技能发现——AI 怎么知道有哪些技能可用？

```
  用户的消息: "帮我提交一下代码"
       │
       ▼
  技能发现系统检查:
  "这条消息涉及到的技能有哪些？"
       │
       ▼
  在系统提示词中提示 AI:
  "你有一个 /commit 技能可以使用，
   它可以智能生成 commit message 并提交"
       │
       ▼
  AI 决定: "我应该用 Skill 工具调用 /commit"
       │
       ▼
  SkillTool 执行 → commit 技能的提示词加载
  → AI 按提示词完成 git commit
```

### 内置技能一览

```
  ┌────────────── 内置技能 ──────────────┐
  │                                      │
  │  /commit      智能 Git 提交           │
  │  /review      代码审查                │
  │  /simplify    代码简化和重构           │
  │  /remember    保存记忆到 CLAUDE.md     │
  │  /claude-api  Claude API 开发辅助     │
  │  /debug       调试辅助                │
  │  /verify      实现验证                │
  │  /loop        周期性执行              │
  │  /update-config 更新配置              │
  │                                      │
  │  + 插件可以注册更多技能               │
  │  + 用户可以自定义技能                  │
  └──────────────────────────────────────┘
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/skills/` | 技能定义目录 |
| `src/tools/SkillTool/` | AI 调用技能的工具 |
| `src/utils/skills/` | 技能发现、缓存、调用逻辑 |

## 模块 12：UI 层 (Terminal User Interface)

### 这是什么？

UI 层负责你在终端里看到的所有界面——消息显示、输入框、权限确认对话框、进度条、代理任务追踪。它使用 **React + Ink** 构建，这意味着终端 UI 是用和写网页类似的方式开发的。

### 用一个比喻来理解

```
  传统终端程序:
  就像打字机 —— 一行一行往下打，打完就没法改了

  Claude Code 的终端 UI:
  就像一个"终端里的 App" —— 有按钮、有布局、有状态、实时更新
  （虽然是纯文字，但有丰富的交互和布局）
```

### 为什么用 React 写终端 UI？

```
  React 的核心理念: UI = f(state)
  "界面是状态的函数，状态变了界面自动更新"

  在网页里:
  React → DOM → 浏览器渲染

  在终端里:
  React → Ink → ANSI 字符 → 终端渲染

  好处:
  • 开发范式一样（组件化、状态管理、生命周期）
  • 能用 React 生态的所有编程模式
  • 组件可复用（消息组件、权限组件等）
```

### UI 层的组件架构

## 模块 13：IDE 集成 (Bridge System)

### 这是什么？

Bridge 系统让 Claude Code 能作为 VS Code 或 JetBrains 的"后端"运行——IDE 扩展提供图形界面，Bridge 负责双向通信。

### 用一个比喻来理解

```
  没有 Bridge:
  你 ←→ 终端 ←→ Claude Code
  （纯终端体验）

  有了 Bridge:
  你 ←→ VS Code 界面 ←→ Bridge ←→ Claude Code
  （图形界面体验，但核心是同一个 Claude Code）
```

就像一个翻译服务：
- **IDE 扩展** = 前台客服（漂亮的图形界面）
- **Bridge** = 后台翻译（把 IDE 的请求翻译成 Claude Code 能理解的格式）
- **Claude Code** = 真正干活的人

### Bridge 怎么工作？

```
  ┌────────────── VS Code ─────────────────┐
  │                                         │
  │  Claude Code 扩展面板                    │
  │  ┌───────────────────────────────┐     │
  │  │  用户在 VS Code 里输入消息     │     │
  │  │  "帮我重构这个函数"            │     │
  │  └──────────────┬────────────────┘     │
  │                 │                      │
  └─────────────────┼──────────────────────┘
                    │ HTTP Polling
                    ▼
  ┌────────────── Bridge ──────────────────┐
  │                                         │
  │  bridgeMain.ts                          │
  │  ┌─────────────────────────────────┐   │
  │  │ 1. 从 IDE 获取工作请求           │   │
  │  │ 2. 启动 Session 子进程           │   │
  │  │ 3. 流式返回执行结果              │   │
  │  │ 4. 权限请求中转给 IDE UI         │   │
  │  └──────────────┬──────────────────┘   │
  │                 │                      │
  └─────────────────┼──────────────────────┘
                    │
                    ▼
  ┌────────── Session 子进程 ───────────────┐
  │                                         │
  │  一个独立的 Claude Code 实例             │
  │  和终端模式功能完全一样                   │
  │  只是输入输出通过 Bridge 中转             │
  │                                         │
  └─────────────────────────────────────────┘
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/bridge/bridgeMain.ts` | Bridge 主循环 |
| `src/bridge/bridgeMessaging.ts` | 消息协议 |
| `src/bridge/sessionRunner.ts` | 会话管理 |
| `src/bridge/bridgePermissionCallbacks.ts` | 权限中转 |

## 模块 14：持久化记忆 (Memory System)

### 这是什么？

记忆系统让 Claude Code 能在不同对话之间"记住"关于你和项目的信息——你的偏好、项目规则、上次讨论的决定。

### 用一个比喻来理解

想象你有一个私人秘书（Claude），但每次见面秘书都会失忆。记忆系统就像给秘书一个**笔记本**，让 TA 每次见面前先翻笔记本，回忆你的偏好。

```
  没有记忆系统:
  第 1 次: "我喜欢简洁的回复"
  第 2 次: "我喜欢简洁的回复" ← 又得说一遍
  第 3 次: "我喜欢简洁的回复" ← 又得说一遍...

  有了记忆系统:
  第 1 次: "我喜欢简洁的回复" → 记到笔记本里
  第 2 次: (自动从笔记本读取) → 直接给出简洁回复
  第 3 次: (自动从笔记本读取) → 直接给出简洁回复 ✨
```

### 记忆存在哪里？

```
  ~/.claude/projects/{项目路径}/memory/
  │
  ├── MEMORY.md          ← 记忆索引（目录）
  │                         最多 200 行
  │
  ├── user_role.md       ← 用户记忆：你的角色和偏好
  ├── feedback_style.md  ← 反馈记忆：你教过 AI 什么
  ├── project_auth.md    ← 项目记忆：项目背景和进展
  └── reference_docs.md  ← 引用记忆：外部资源指针
```

### 四种记忆类型

```
  ┌────────── user（用户记忆）──────────┐
  │                                     │
  │  记什么: 你的角色、专长、偏好          │
  │  例如: "用户是高级后端工程师，         │
  │        熟悉 Go 但不熟悉 React"       │
  │                                     │
  │  用途: 调整回答的深度和角度            │
  └─────────────────────────────────────┘

  ┌────────── feedback（反馈记忆）──────┐
  │                                     │
  │  记什么: 你纠正过 AI 的行为           │
  │  例如: "不要在回复末尾做总结，          │
  │        用户说过看 diff 就够了"        │
  │                                     │
  │  用途: 避免重复犯同样的错误            │
  └─────────────────────────────────────┘

  ┌────────── project（项目记忆）───────┐
  │                                     │
  │  记什么: 项目的背景信息               │
  │  例如: "认证模块重写是因为合规要求，    │
  │        不是技术债务清理"              │
  │                                     │
  │  用途: 理解为什么做这个改动            │
  └─────────────────────────────────────┘

  ┌────────── reference（引用记忆）─────┐
  │                                     │
  │  记什么: 外部系统的指针               │
  │  例如: "Bug 追踪在 Linear 的          │
  │        INGEST 项目里"                │
  │                                     │
  │  用途: 知道去哪找相关信息              │
  └─────────────────────────────────────┘
```

### 关键文件

## 模块 15：多代理协调 (Coordinator)

### 这是什么？

Coordinator 模式让一个 Claude 实例（"领导"）指挥多个 Claude 子实例（"工人"）并行工作。

### 用一个比喻来理解

```
  单代理模式 (默认):
  ┌──────────────┐
  │  一个人干活    │  → 一次只能做一件事
  │  (Claude)     │
  └──────────────┘

  多代理协调模式:
  ┌──────────────┐
  │  项目经理      │  → 分配任务、综合结果
  │  (Coordinator) │
  └──────┬───────┘
         │ 分配任务
    ┌────┼────┬────────┐
    ▼    ▼    ▼        ▼
  ┌────┐┌────┐┌────┐┌────┐
  │工人1││工人2││工人3││工人4│  → 并行干活！
  │研究 ││研究 ││写代码││测试 │
  └────┘└────┘└────┘└────┘
```

### 协调策略

```
  Coordinator 的工作方式:

  阶段 1: 并行研究
  ┌─────────────────────────────────────┐
  │  "每人去调研一个模块的实现方案"        │
  │                                     │
  │  工人 A → 研究 auth 模块             │
  │  工人 B → 研究 database 模块         │  同时进行！
  │  工人 C → 研究 API 模块              │
  └─────────────────────────────────────┘
                   │
                   ▼
  阶段 2: 综合分析
  ┌─────────────────────────────────────┐
  │  Coordinator 收集所有研究结果          │
  │  制定统一的实现方案                    │
  └─────────────────────────────────────┘
                   │
                   ▼
  阶段 3: 顺序实现
  ┌─────────────────────────────────────┐
  │  "涉及同一个文件区域的修改按顺序做"    │
  │                                     │
  │  工人 A → 改 auth (先做)             │
  │  等完成后...                          │
  │  工人 B → 改 database (再做)          │
  │  (避免冲突)                          │
  └─────────────────────────────────────┘
                   │
                   ▼
  阶段 4: 验证
  ┌─────────────────────────────────────┐
  │  工人 D → 跑测试，验证一切正常         │
  └─────────────────────────────────────┘
```

| 文件 | 作用 |
|------|------|
| `src/coordinator/coordinatorMode.ts` | 协调器模式开关和配置 |
| `src/tools/AgentTool/` | 子代理启动逻辑 |
| `src/tools/SendMessageTool/` | 代理间消息传递 |

## 模块 16：遥测分析 (Analytics)

### 这是什么？

遥测分析系统收集 Claude Code 的使用数据——哪些工具用得多、性能如何、出了什么错——帮助 Anthropic 改进产品。

### 它不是什么？

```
  ⚠️  重要说明:
  
  ✅ 收集: 工具使用频率、错误类型、性能指标
  ❌ 不收集: 你的代码内容、文件路径、个人信息
  
  有专门的 marker type 防止敏感数据泄露:
  AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS
  （类型名就是在提醒开发者："我确认这不是代码或文件路径"）
```

### 遥测架构

```
  ┌────── 事件产生 ──────┐
  │                      │
  │  工具被调用           │
  │  权限被批准/拒绝      │
  │  API 调用完成         │
  │  错误发生             │
  │                      │
  └──────────┬───────────┘
             │ logEvent()
             ▼
  ┌────── 事件队列 ──────┐
  │                      │
  │  事件先存在队列里      │
  │  等后端就绪再发送      │
  │                      │
  └──────────┬───────────┘
             │ attachAnalyticsSink()
             ▼
  ┌────── 事件路由 ──────┐
  │                      │
  │  ┌───────────────┐   │
  │  │  Datadog      │   │  → 性能监控
  │  └───────────────┘   │
  │  ┌───────────────┐   │
  │  │  1P Exporter  │   │  → 内部分析
  │  └───────────────┘   │
  │  ┌───────────────┐   │
  │  │  GrowthBook   │   │  → 实验和功能开关
  │  └───────────────┘   │
  │                      │
  └──────────────────────┘

  用户可以关闭:
  • 设置 privacy level = no-telemetry
  • 使用第三方 API (Bedrock/Vertex 自动关闭)
```
