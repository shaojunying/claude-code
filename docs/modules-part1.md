## 模块 1：入口模块 (main.tsx)

### 这是什么？

`main.tsx` 就是整个程序的"大门"。当你在终端里敲下 `claude` 命令的时候，最先运行的就是这个文件。

### 用一个比喻来理解

想象你走进一家餐厅：
- **门口接待员**（main.tsx）：问你几位、有没有预约、想坐包间还是大厅
- 然后根据你的回答，把你带到不同的区域

`main.tsx` 就是这个"接待员"——它解析你的命令行参数，然后把你引导到正确的运行模式。

### 它具体做什么？

```
你在终端敲: claude --model opus "帮我写个函数"
                    │              │
                    │              └── 你的问题（输入内容）
                    └── 命令行参数（选项）

                         │
                         ▼

            ┌─────────────────────────┐
            │      main.tsx 做的事     │
            │                         │
            │  1. 并行预取 (抢跑)      │

   │REPL    │  │(脚本调用)│  │ 模式   │   │ 模式     │

### 关于"并行预取"——为什么要抢跑？

这是一个性能优化技巧。打个比方：

- **没有预取**：你先穿鞋（65ms），再拿钥匙（30ms），再出门 = 串行等待
- **有预取**：你一边穿鞋，一边让朋友帮你拿钥匙 = 并行执行

```
没有预取（串行 ~230ms）:
  ├─ 加载模块 ─────────────┤
                            ├─ 读密钥 ──────┤
                                            ├─ 读策略 ───┤
                                                         → 启动完成

有预取（并行 ~135ms）:
  ├─ 加载模块 ─────────────┤
  ├─ 读密钥 ──────┤         → 启动完成（快了约 40%！）
  ├─ 读策略 ───┤
```

### 它和谁打交道？

```
main.tsx
  │
  ├──→ Commander.js    (解析命令行参数)
  ├──→ React/Ink       (启动终端 UI)
  ├──→ QueryEngine     (创建对话引擎)
  ├──→ 配置系统         (读取用户设置)
  └──→ Bridge          (如果是 IDE 模式)
```

## 模块 2：对话引擎 (QueryEngine)

### 这是什么？

`QueryEngine` 是整个 Claude Code 的"大脑"。它管理着你和 AI 之间的整个对话过程——记住你说了什么、AI 回复了什么、执行了哪些工具、花了多少 Token。

### 用一个比喻来理解

想象一个**翻译官**在你和一个外国专家之间来回沟通：

- 你说中文（用户输入）
- 翻译官把你的话+背景资料一起翻译给专家（构建提示词 → 调用 API）
- 专家有时候需要查资料（工具调用）
- 翻译官帮专家去查，拿回结果再转述给专家
- 最后翻译官把专家的回答翻译给你

`QueryEngine` 就是这个"翻译官"。

## 模块 3：查询管道 (query.ts)

### 这是什么？

如果 `QueryEngine` 是"大脑"，那 `query.ts` 就是"神经系统"——它负责实际的数据传输：把请求发给 API、接收流式响应、执行工具调用、处理错误。

### 用一个比喻来理解

想象一条**流水线**：

```
  原材料投入     加工处理      质检       包装出库
  (用户输入)  (API 流式响应) (权限检查) (展示结果)
     │            │            │          │
     ▼            ▼            ▼          ▼
  ═══════════════════════════════════════════  ← 传送带（AsyncGenerator）
```

`query()` 函数是一个 `AsyncGenerator`（异步生成器），就像一条传送带——数据从一端进去，一块一块地从另一端出来，不需要等全部处理完。

### 它管理哪些事情？

```
┌──────────────────── query() 管道 ─────────────────────┐
│                                                        │
│  📡 流式传输                                            │
│  ├── API 响应一个字一个字地传回来（不是一次性返回）        │
│  ├── 你能实时看到 AI "正在打字"                          │
│  └── 就像微信聊天时看到对方"正在输入..."                  │
│                                                        │
│  🔧 工具执行                                            │
│  ├── AI 说"我需要读文件" → 立刻去读                      │
│  ├── 多个互不影响的工具可以同时跑（并行）                  │
│  └── 有依赖的工具按顺序跑（串行）                         │
│                                                        │
│  📦 上下文压缩                                          │
│  ├── 对话太长了？自动总结历史消息                         │
│  ├── 就像老师说"总结前面讲的要点"                        │
│  └── 确保不超过模型的"记忆上限"                          │
│                                                        │
│  🔒 权限检查                                            │
│  ├── 每个工具调用前先问："允许这么做吗？"                  │
│  ├── 有些操作自动允许（读文件）                           │
│  └── 有些操作需要你点确认（删文件）                       │
│                                                        │
│  💰 预算控制                                            │
│  ├── Token 用了多少  （不能无限花钱）                     │
│  ├── 对话轮数上限    （不能无限循环）                     │
│  └── 成本预算        （不能超预算）                      │
│                                                        │
│  📎 附件注入                                            │
│  ├── CLAUDE.md 内容   （项目规则）                       │
│  └── 记忆文件         （上次的笔记）                     │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 关于"上下文压缩"——为什么需要？

AI 模型有"记忆上限"（上下文窗口），就像人脑一样不能记住无限多的东西：

```
对话刚开始时:
┌─────────────────────────────────────┐
│ 系统提示词    ████                   │
│ 用户消息 1    ██                     │
│ AI 回复 1     ████                   │
│ 用户消息 2    ██                     │
│                                     │
│              ... 还有很多空间 ...     │
│                                     │
│                     上下文窗口上限 →  │
└─────────────────────────────────────┘

对话进行了很久后:
┌─────────────────────────────────────┐
│ 系统提示词    ████                   │
│ 用户消息 1-5  ████████████████████  │
│ AI 回复 1-5   ████████████████████  │
│ 工具结果 x10  ████████████████████  │
│ 用户消息 6    ██                    │ ← 快满了！
│                     上下文窗口上限 →  │
└─────────────────────────────────────┘

压缩之后:
┌─────────────────────────────────────┐
│ 系统提示词    ████                  │
│ [历史摘要]    ████████              │  ← 把前面的浓缩成摘要
│ 最近消息      ████████████          │
│                                     │
│              ... 又有空间了 ...       │
│                                     │
│                     上下文窗口上限 →  │
└─────────────────────────────────────┘
```

### 它和谁打交道？

```
  QueryEngine (调用方)
       │
       ▼
    query() ←──── 从 Claude API 获取流式响应
       │
       ├──→ StreamingToolExecutor (执行工具)
       ├──→ Permission Gateway (检查权限)
       ├──→ Compaction (压缩历史)
       └──→ Cost Tracker (追踪成本)
```

## 模块 4：工具系统 (Tool System)

### 这是什么？

工具系统赋予了 Claude 操作真实世界的能力。模型本身只能"思考和说话"，而工具让它能"动手干活"——读文件、写代码、跑命令、搜索网页。

### 用一个比喻来理解

Claude AI 就像一个远程的**建筑师**：
- 它能画图纸、做规划（思考和回复）
- 但它没有手，不能自己砌墙

**工具**就是给建筑师配的各种**工人**：
- 👷 **Bash 工具** = 能执行任何终端命令的"万能工人"
- 📖 **FileRead 工具** = 专门查看图纸的"档案员"
- ✏️ **FileEdit 工具** = 修改图纸的"绘图员"
- 🔍 **Grep 工具** = 在文件堆里找关键词的"搜索员"

### 一个工具长什么样？

每个工具都是一个符合统一接口的对象。以 `GlobTool`（文件搜索）为例：

```
┌────────────────── GlobTool ──────────────────┐
│                                              │
│  name: "Glob"              ← 工具名字        │
│  description: "按模式查找文件" ← 给 AI 看的说明│
│                                              │
│  inputSchema:              ← 输入参数定义     │
│  {                                           │
│    pattern: "**/*.ts"      ← 搜索模式        │
│    path: "/src"            ← 在哪个目录搜     │
│  }                                           │
│                                              │
│  call(input, context):     ← 实际执行逻辑     │
│    → 调用 ripgrep 搜索                        │
│    → 返回匹配的文件列表                        │
│                                              │
│  isConcurrencySafe: true   ← 可以并行执行     │
│  isReadOnly: true          ← 不修改任何东西    │
│                                              │
│  renderToolUseMessage:     ← 执行时的 UI      │
│    → "🔍 正在搜索 **/*.ts..."                │
│                                              │
│  renderToolResultMessage:  ← 结果的 UI        │
│    → "找到 42 个文件"                         │
│                                              │
└──────────────────────────────────────────────┘
```

### 工具的并行执行——为什么要分组？

AI 经常一次请求多个工具调用。有些工具可以同时运行（互不影响），有些必须排队：

```
AI 请求: "读文件 A、读文件 B、然后编辑文件 A"

           ┌──────────────── StreamingToolExecutor ────────────────┐
           │                                                      │
           │  第一组（并发安全，同时跑）:                             │
           │  ┌──────────────┐  ┌──────────────┐                  │
           │  │  读文件 A     │  │  读文件 B     │   ← 互不影响     │
           │  │  (FileRead)  │  │  (FileRead)  │   ← 同时执行！   │
           │  └──────┬───────┘  └──────┬───────┘                  │
           │         │                 │                           │
           │         └────────┬────────┘                          │
           │                  │ 都完成了                            │
           │                  ▼                                    │
           │  第二组（需要排队）:                                    │
           │  ┌──────────────┐                                    │
           │  │  编辑文件 A   │   ← 必须等读文件完成后               │
           │  │  (FileEdit)  │   ← 才能安全编辑                    │
           │  └──────────────┘                                    │
           │                                                      │
           └──────────────────────────────────────────────────────┘

为什么要这样做？

  想象一下：如果"读文件 A"和"编辑文件 A"同时执行会怎样？
  → 可能读到的是编辑前的内容，也可能是编辑后的
  → 结果不确定，会导致 bug！

  所以：读-读可以并行，读-写必须串行。
```

## 模块 6：权限系统 (Permission System)

### 这是什么？

权限系统是 Claude Code 的"安全卫士"。它确保 AI 不会在你不知情的情况下做危险的事——比如删除文件、执行危险命令、修改系统配置。

### 用一个比喻来理解

想象你雇了一个非常能干的助手（Claude），但你得设置"权限级别"：

```
  🟢 完全信任模式 (bypassPermissions)
     "你想干啥就干啥，不用问我"
     → 适合你完全信任的操作环境

  🟡 默认模式 (default)  ← 大多数人用这个
     "读文件可以，但改文件前要先问我"
     → 安全和效率的平衡

  🔴 严格模式 (plan)
     "你只能看、不能动"
     → 适合敏感项目或学习阶段
```

### 权限模式详解

```
┌─ 权限模式从宽松到严格 ─────────────────────────────────────────┐
│                                                               │
│  bypassPermissions   ████████████████████████  全部自动批准    │
│  "啥都行"                                                     │
│                                                               │
│  acceptEdits         ████████████████████      自动应用编辑    │
│  "改文件不用问我"                                               │
│                                                               │
│  auto                ████████████████         ML 自动判断     │
│  "AI 来判断该不该问"    (实验性功能)                              │
│                                                               │
│  default             ████████████            敏感操作问用户    │
│  "改东西先问我"        ← 推荐模式                               │
│                                                               │
│  plan                ████████                只读模式          │
│  "只能看不能改"                                                 │
│                                                               │
│  dontAsk             ████                    默认拒绝          │
│  "什么都不让干"                                                 │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 权限检查流程——AI 要执行操作时发生了什么？

```
  AI: "我要执行 rm -rf node_modules"
                │
                ▼
  ┌───────────────────────────────┐
  │  第一关: Hook 预检查           │
  │                               │
  │  有没有提前配置的自动化规则？   │
  │  比如: "npm 相关命令自动允许"  │
  │                               │
  │  结果:                         │
  │    自动批准 → ✅ 直接执行      │
  │    自动拒绝 → ❌ 直接阻止      │
  │    没匹配   → 继续往下 ↓       │
  └─────────────┬─────────────────┘
                │
                ▼
  ┌───────────────────────────────┐
  │  第二关: 规则匹配              │
  │                               │
  │  检查配置文件中的规则:          │
  │                               │
  │  alwaysAllow:                 │
  │    - "Bash(npm *)"  ← 匹配？  │
  │  alwaysDeny:                  │
  │    - "Bash(rm -rf *)" ← 匹配！│
  │                               │
  │  → 被 alwaysDeny 拦住了 ❌     │
  └─────────────┬─────────────────┘
                │
                ▼ (如果规则没匹配到)
  ┌───────────────────────────────┐
  │  第三关: 问用户                │
  │                               │
  │  ┌─────────────────────────┐  │
  │  │ Claude 想要执行:         │  │
  │  │                         │  │
  │  │ rm -rf node_modules     │  │
  │  │                         │  │
  │  │ [✅ 允许]  [❌ 拒绝]     │  │
  │  │ [✅ 始终允许]            │  │
  │  └─────────────────────────┘  │
  │                               │
  │  用户选择 → 记录决定           │
  └───────────────────────────────┘
```

### 权限规则的优先级——谁说了算？

多个地方都可以设置规则，但它们有优先级：

```
  最高优先级 (铁律，不可覆盖)
       │
  ┌────┴───────────────────────┐
  │ ① 命令行参数                │  你启动时传的 --permission-mode
  ├────────────────────────────┤
  │ ② 组织策略 (MDM)           │  公司 IT 部门统一设定的
  ├────────────────────────────┤
  │ ③ Feature Flag             │  功能标志覆盖
  ├────────────────────────────┤
  │ ④ 项目设置                  │  .claude/settings.json
  │   (团队共享)                │  提交到 Git，团队共用
  ├────────────────────────────┤
  │ ⑤ 用户设置                  │  ~/.claude/settings.json
  │   (个人偏好)                │  你自己的默认偏好
  ├────────────────────────────┤
  │ ⑥ 会话设置                  │  当前会话临时规则
  │   (临时的)                  │  关掉就没了
  └────────────────────────────┘
       │
  最低优先级

  举例:
  如果公司策略说 "禁止执行 rm -rf"，
  那即使你自己设置了 "允许 rm -rf"，
  也会被公司策略覆盖 → 仍然禁止。
```

### 它和谁打交道？

```
  ┌──────────────┐
  │ 配置系统      │──→ 提供权限规则
  └──────────────┘
         │
         ▼
  ┌──────────────┐     ┌──────────────┐
  │ 工具系统      │──→  │ 权限系统      │  ←── 用户 (点允许/拒绝)
  │ "我要执行XX"  │     │ "准许/拒绝"   │
  └──────────────┘     └──────┬───────┘
                              │
                         ┌────┴────┐
                         ▼         ▼
                    ✅ 执行工具  ❌ 返回拒绝信息
```

| `src/hooks/toolPermission/PermissionContext.ts` | 权限决策引擎核心 |
| `src/hooks/toolPermission/permissionLogging.ts` | 权限决策日志 |
| `src/hooks/toolPermission/handlers/interactiveHandler.ts` | REPL 交互式权限处理 |

## 模块 7：配置系统 (Configuration)

### 这是什么？

配置系统管理 Claude Code 的所有设置——用哪个模型、允许哪些工具、MCP 服务器配置、主题颜色等等。

### 用一个比喻来理解

就像手机的"设置"App，但有多层覆盖：

- **系统默认**：手机出厂设置
- **用户设置**：你自己改的偏好（铃声、壁纸）
- **公司策略**：如果是公司手机，IT 部门可以强制某些设置
- **临时设置**：开会时临时静音

Claude Code 也是一样，配置有多个来源，后面的可以覆盖前面的。

### 配置来源层级

```
  ┌──────────────────── 配置来源（从低到高）─────────────────────┐
  │                                                             │
  │  优先级最低 ← ← ← ← ← ← ← ← ← ← ← ← ← ← → 优先级最高  │
  │                                                             │
  │   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐      │
  │   │ 用户设置 │→│ 项目设置 │→│ 本地设置 │→│ 组织策略 │      │
  │   │         │  │         │  │         │  │         │      │
  │   │~/.claude│  │.claude/ │  │.claude/ │  │ MDM /   │      │
  │   │settings │  │settings │  │settings │  │ 注册表  │      │
  │   │.json   │  │.json   │  │.local   │  │         │      │
  │   │         │  │         │  │.json   │  │         │      │
  │   │ (你的   │  │ (团队   │  │ (你本地 │  │ (公司IT │      │
  │   │  个人   │  │  共享   │  │  独有的 │  │  强制)  │      │
  │   │  偏好)  │  │  约定)  │  │  覆盖)  │  │         │      │
  │   └─────────┘  └─────────┘  └─────────┘  └─────────┘      │
  │                                                             │
  │                             继续覆盖 ↓                      │
  │                                                             │
  │   ┌─────────────┐  ┌────────────┐  ┌──────────────┐       │
  │   │ 托管配置     │→│ Feature Flag│→│ CLI 命令行参数 │       │
  │   │             │  │            │  │              │       │
  │   │ managed-    │  │ --flag-    │  │ --model      │       │
  │   │ settings    │  │ settings   │  │ --permission │       │
  │   │ .json       │  │ FILE       │  │ -mode        │       │
  │   │             │  │            │  │              │       │
  │   │ (集中管理)   │  │ (功能开关)  │  │ (最终覆盖)   │       │
  │   └─────────────┘  └────────────┘  └──────────────┘       │
  │                                          ↑                 │
  │                                     优先级最高！             │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

  实际意义:
  - 你在 ~/.claude/settings.json 设了 model: "sonnet"

## 模块 8：状态管理 (State Management)

### 这是什么？

状态管理负责保存和更新程序运行时的所有"变量"——当前用哪个模型、花了多少钱、连接了哪些 MCP 服务器、用户设置了什么主题等等。

### 用一个比喻来理解

就像一个电子游戏的"存档系统"：
- **Bootstrap State**（全局状态）= 游戏的"元数据"——当前关卡、总分、存档时间
- **AppState**（应用状态）= 游戏的"实时状态"——角色位置、血量、背包物品

### 两层状态管理

```
┌─────────────── 状态管理的两层架构 ────────────────┐
│                                                   │
│  第一层: Bootstrap State (全局单例)                │
│  ┌─────────────────────────────────────────┐     │
│  │  位置: src/bootstrap/state.ts            │     │
│  │  特点: 整个程序只有一份                    │     │
│  │                                         │     │
│  │  存什么:                                 │     │
│  │  • 会话 ID（"这次对话的身份证"）           │     │
│  │  • 项目根目录（"在哪个项目里工作"）        │     │
│  │  • 总花费（"一共花了多少钱"）             │     │
│  │  • 模型用量统计（"每个模型用了多少 Token"）│     │
│  │  • 错误日志（"出过哪些错"）               │     │
│  │                                         │     │
│  │  访问方式:                               │     │
│  │  getSessionId() / setSessionId()        │     │
│  │  getTotalCostUSD() / incrTotalCostUSD() │     │
│  │  （getter / setter 对，很简单）            │     │
│  └─────────────────────────────────────────┘     │
│                                                   │
│  第二层: AppState (响应式状态)                      │
│  ┌─────────────────────────────────────────┐     │
│  │  位置: src/state/                        │     │
│  │  特点: 变化时自动通知 UI 更新              │     │
│  │                                         │     │
│  │  存什么:                                 │     │
│  │  • 用户设置（"偏好什么模式"）              │     │
│  │  • MCP 连接（"连了哪些服务器"）           │     │
│  │  • 插件状态（"装了哪些插件"）              │     │
│  │  • 任务列表（"有哪些进行中的任务"）        │     │
│  │  • UI 状态（"展开还是折叠"）              │     │
│  │                                         │     │
│  │  访问方式:                               │     │
│  │  getAppState()                          │     │
│  │  setAppState(prev => newState)          │     │
│  │  subscribe(listener)  ← 状态变了自动通知  │     │
│  └─────────────────────────────────────────┘     │
│                                                   │
│  区别:                                            │
│  • Bootstrap = 基础设施级别（怎么启动、花了多少钱）  │
│  • AppState  = 功能级别（用户在干什么、界面长啥样）  │
│                                                   │
└───────────────────────────────────────────────────┘
```

### 为什么要分两层？

```
  想象一个公司:

  Bootstrap State = 公司的"基本信息"
  ┌──────────────────────────────────┐
  │  公司名称: Claude Code Inc.      │
  │  成立日期: (sessionId)           │
  │  办公地址: (projectRoot)         │
  │  财务报表: (totalCostUSD)        │
  │                                  │
  │  特点: 很少变化，改了影响全局      │
  └──────────────────────────────────┘

  AppState = 公司的"日常运营"
  ┌──────────────────────────────────┐
  │  今天谁在上班: (tasks)            │
  │  会议室谁在用: (mcp.clients)      │
  │  外卖到了没有: (plugins.status)   │
  │  前台显示屏: (UI state)           │
  │                                  │
  │  特点: 频繁变化，变了 UI 要更新    │
  └──────────────────────────────────┘
```

### 响应式更新——状态变了会怎样？

```
  setAppState(prev => ({
    ...prev,
    theme: "light"    ← 用户切换了主题
  }))
      │
      ▼
  ┌──────────────────────────┐
  │  Store 检测到变化          │
  │  通知所有订阅者            │
  └──────────┬───────────────┘
             │
     ┌───────┼───────┬───────────┐
     ▼       ▼       ▼           ▼
  UI 组件  API 客户端  配置持久化  遥测日志
  重新渲染  更新请求头  写入磁盘    记录事件
```

### 它和谁打交道？

```
  几乎所有模块都要读/写状态:

  ┌──────────────────────────────────────────────┐
  │              State Management                 │
  │                                              │
  │  读取方 (getAppState / getXXX):               │
  │  • 所有 UI 组件     → 决定显示什么             │
  │  • QueryEngine      → 知道当前设置             │
  │  • 权限系统          → 知道当前权限模式         │
  │  • 工具系统          → 知道哪些工具已启用       │
  │                                              │
  │  写入方 (setAppState / setXXX):               │
  │  • 用户操作          → 改设置、切主题           │
  │  • API 调用          → 更新 Token 用量         │
  │  • MCP 连接          → 更新连接状态            │
  │  • 任务系统          → 更新任务进度            │
  │                                              │
  └──────────────────────────────────────────────┘
```
