## 专题 1：错误处理与重试机制

### 问题是什么？

Claude Code 需要通过网络调用 Claude API，这意味着随时可能遇到：
- **限流 (429)**：你请求太频繁了
- **过载 (529)**：服务器太忙了
- **网络错误**：连接断了、超时了
- **认证失败**：Token 过期了
- **内容太大**：上传的图片/文件超限

### 重试策略——指数退避 + 抖动

```
  第 1 次失败 → 等 500ms + 随机抖动
  第 2 次失败 → 等 1000ms + 随机抖动
  第 3 次失败 → 等 2000ms + 随机抖动
  第 4 次失败 → 等 4000ms + 随机抖动
  ...
  上限: 等 32000ms (32秒)
  总共最多重试 10 次

  为什么加"随机抖动"？
  ┌─────────────────────────────────────────┐
  │ 想象 1000 个用户同时被限流，              │
  │ 如果大家都等恰好 1 秒后重试，              │
  │ 服务器会在 1 秒后再次被冲垮。              │
  │                                         │
  │ 加了随机抖动后，1000 个用户分散在           │
  │ 0.5~1.5 秒之间重试，服务器压力分散。        │
  └─────────────────────────────────────────┘
```

### 智能降级——连续 3 次过载自动换模型

```
  连续 3 次 529 错误
       │
       ▼
  ┌─────────────────────────────────┐
  │ 当前模型太繁忙了！                │
  │ 自动切换为 fallback 模型         │
  │                                 │
  │ 例如: opus → sonnet (更轻量，    │
  │       但不那么"聪明")            │
  │                                 │
  │ 通知用户: "由于服务繁忙，         │
  │ 已临时切换到 Sonnet 模型"        │
  └─────────────────────────────────┘
```

### 特殊错误处理

```
  ┌─────── 错误分类处理 ────────┐
  │                             │
  │  限流 (429)                  │
  │  ├── Pro/Max 用户: 不重试    │
  │  │   (尊重用户的配额限制)     │
  │  └── 其他用户: 正常重试       │
  │                             │
  │  内容太大                    │
  │  ├── 图片太大: 友好提示压缩   │
  │  └── 提示词太长: 触发上下文   │
  │      压缩 (见专题 2)         │
  │                             │
  │  SSL/TLS 错误               │
  │  ├── 深度解析错误链 (5 层)    │
  │  └── 可能是企业代理问题       │
  │                             │
  │  连接重置 (ECONNRESET)       │
  │  ├── 禁用 keep-alive        │
  │  └── 强制重建连接            │
  │                             │
  └─────────────────────────────┘
```

## 专题 2：上下文压缩 (Compaction)

### 问题是什么？

AI 模型有"记忆上限"（上下文窗口）。对话越长，消耗越多。当快要超限时，需要"压缩"历史消息，只保留重要信息。

### 压缩什么时候触发？

```
  ┌─────── 三种触发方式 ────────┐
  │                             │
  │  ① 手动触发                  │
  │  用户输入 /compact           │
  │                             │
  │  ② 定时触发                  │
  │  每 N 轮对话自动检查          │
  │  (可配置)                    │
  │                             │
  │  ③ 响应式触发                │
  │  API 报错"提示词太长"时       │
  │  自动逐步压缩                │
  └─────────────────────────────┘
```

### 响应式压缩——逐步降级策略

当 API 报"提示词太长"错误时，系统不会一次性大幅压缩，而是逐步尝试：

```
  第一轮: 移除图片
  ┌────────────────────────────────────┐
  │ 把对话中的图片替换成文字标记:        │
  │ [此处原有一张截图]                   │
  │                                    │
  │ 够不够？ → 够了就停 / 不够继续 ↓     │
  └────────────────────────────────────┘

  第二轮: 移除所有媒体内容
  ┌────────────────────────────────────┐
  │ 移除 PDF、音频等附件                │
  │                                    │
  │ 够不够？ → 够了就停 / 不够继续 ↓     │
  └────────────────────────────────────┘

  第三轮: 压缩消息组
  ┌────────────────────────────────────┐
  │ 把多轮对话总结成一段摘要:            │
  │                                    │
  │ 之前: 10 条消息 (用户问题 + AI 回复  │
  │       + 工具调用 + 工具结果 ×5)      │
  │                                    │
  │ 之后: 1 段摘要                      │
  │ "用户询问了排序算法，我读取了        │
  │  utils.ts 并修改了第 42-58 行..."   │
  └────────────────────────────────────┘
```

### 压缩后的"善后"工作

```
  压缩完成后不是就完事了，还要:

  ┌─────────────────────────────────────┐
  │ 1. 重新注入最近修改的文件 (最多 5 个)  │
  │    因为压缩可能丢失了文件的最新状态     │
  │    每个文件最多 50KB                   │
  │                                       │
  │ 2. 重新注入相关技能 (最多 5 个)         │
  │    确保 AI 还记得有哪些技能可用         │
  │    每个技能最多 5KB                    │
  │                                       │
  │ 3. 执行 post-compact 钩子              │
  │    通知用户配置的自动化脚本             │
  └─────────────────────────────────────┘
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/services/compact/` | 压缩系统主目录 |
| `src/services/compact/grouping.ts` | 消息分组逻辑 |

## 专题 3：会话持久化与恢复

### 问题是什么？

你和 Claude 聊了很久，突然要关掉终端去开会。回来后希望能从上次断开的地方继续。

### 会话存在哪里？

```
  ~/.claude/projects/{项目路径哈希}/{sessionId}.jsonl

  文件格式: JSONL (每行一个 JSON 对象)
  ┌──────────────────────────────────────────┐
  │ {"type": "user", "text": "帮我写排序"}    │
  │ {"type": "assistant", "text": "好的..."}  │
  │ {"type": "tool_use", "tool": "FileRead"}  │
  │ {"type": "tool_result", "data": "..."}    │
  │ ...                                       │
  └──────────────────────────────────────────┘

  为什么用 JSONL 而不是 JSON？
  → 追加写入，不需要每次重写整个文件
  → 即使程序崩溃，已写入的记录不会丢失
```

### 恢复流程

```
  你输入: /resume (或 claude --resume)
       │
       ▼
  ┌─────────────────────────────────┐
  │ 1. 找到最近的会话文件             │
  │    按时间排序，列出可恢复的会话    │
  └──────────┬──────────────────────┘
             │
             ▼
  ┌─────────────────────────────────┐
  │ 2. 读取 JSONL 文件               │
  │    重建消息历史                   │
  └──────────┬──────────────────────┘
             │
             ▼
  ┌─────────────────────────────────┐
  │ 3. 恢复关联状态                  │
  │    • 文件修改历史                 │
  │    • Git 归因信息                │
  │    • 任务列表                    │
  │    • 成本统计 (如果 ID 匹配)      │
  └──────────┬──────────────────────┘
             │
             ▼
  ┌─────────────────────────────────┐
  │ 4. 继续对话                      │
  │    就像从未中断过一样             │
  └─────────────────────────────────┘
```

### 输入历史

```
  你之前输入过的命令也有记忆:

  ┌───────────────────────────────────┐
  │  按 ↑ ↓ 键 → 浏览历史输入         │
  │  Ctrl+R    → 搜索历史输入         │
  │                                   │
  │  存储位置:                         │
  │  短文本 (≤1024 字) → 内联存储     │
  │  长文本 (>1024 字) → 文件存储     │
  │    ~/.claude/pastes/{hash}        │
  │                                   │
  │  每个项目最多保存 100 条           │
  │  按时间排序，去重                  │
  └───────────────────────────────────┘
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/utils/sessionStorage.ts` | 会话文件读写 |
| `src/utils/sessionRestore.ts` | 会话状态恢复 |
| `src/history.ts` | 输入历史管理 |
| `src/commands/resume/` | /resume 命令实现 |

### 子代理权限隔离——为什么重要？

```
  想象一个场景:

  主代理 (你在使用):
    权限: 可以读/写本项目的文件

  主代理启动了一个子代理:
    如果不隔离 → 子代理也能读写你的文件
    → 如果子代理被恶意提示词注入？

  隔离后:
    子代理只能使用明确授权的工具
    异步子代理默认不能弹权限对话框
    → 即使出问题也不会影响主代理

  ┌─────────────────── 权限传递规则 ───────────────────┐
  │                                                    │
  │  父代理                    子代理                   │
  │  allowedTools: [A, B, C]  →  最多: [A, B, C]     │
  │                              可以缩小: [A, B]      │
  │                              不能放大: [A, B, C, D] ❌│
  │                                                    │
  │  文件缓存: 克隆一份给子代理（独立副本）               │
  │  状态: 异步子代理的 setAppState 是空操作            │
  │  MCP: 可复用父代理连接 or 创建自己的                │
  │                                                    │
  └────────────────────────────────────────────────────┘
```

## 专题 5：成本追踪系统

### 怎么计费？

```
  ┌────────── Token 类型和成本 ──────────┐
  │                                      │
  │  输入 Token (input)                   │
  │  └── 你发给 AI 的所有文字              │
  │      包括系统提示词、对话历史、工具结果  │
  │                                      │
  │  输出 Token (output)                  │
  │  └── AI 回复的所有文字                 │
  │      包括思考过程、工具调用请求         │
  │                                      │
  │  缓存读取 Token (cache_read)          │
  │  └── 命中缓存的输入 Token              │
  │      比正常输入 Token 便宜很多！        │
  │                                      │
  │  缓存创建 Token (cache_creation)      │
  │  └── 为缓存创建预热                   │
  │      稍贵，但后续请求能省钱            │
  │                                      │
  └──────────────────────────────────────┘

  成本公式:
  总成本 = input × 单价 + output × 单价
         + cache_read × 折扣单价
         + cache_creation × 稍高单价
         + web_search × 搜索单价
```

### 成本跟踪的数据流

```
  每次 API 调用完成
       │
       ▼
  cost-tracker.ts 记录:
  ├── 本次调用的各类 Token 数量
  ├── 按模型分类累加
  └── 计算美元成本
       │
       ▼
  Bootstrap State 更新:
  ├── totalCostUSD += 本次成本
  └── modelUsage[模型名] += 本次用量
       │
       ▼
  持久化到磁盘:
  ~/.claude/global.json
  ├── lastCost: 上次会话成本
  ├── lastModelUsage: 各模型用量
  └── lastSessionId: 关联会话 ID
       │
       ▼
  /cost 命令展示:
  ┌──────────────────────────────────────┐
  │ 📊 Model Usage                       │
  │                                      │
  │ opus-4.6:                            │
  │   Input:  125,430 tokens ($1.88)     │
  │   Output:  32,100 tokens ($2.41)     │
  │   Cache:   89,200 tokens ($0.45)     │
  │                                      │
  │ sonnet-4.6:                          │
  │   Input:   15,000 tokens ($0.05)     │
  │   Output:   8,200 tokens ($0.12)     │
  │                                      │
  │ Total: $4.91                         │
  │ Duration: 12m 34s (API: 8m 50s)      │
  │ Code: +142 / -38 lines               │
  └──────────────────────────────────────┘
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/cost-tracker.ts` | 成本计算和累加逻辑 |
| `src/utils/modelCost.ts` | 各模型的定价表 |
| `src/costHook.ts` | 成本变更回调 |

## 专题 6：Feature Flag 系统

### 这是什么？

Feature Flag（功能开关）让开发者可以选择性地开启或关闭某些功能，而不需要修改代码。Claude Code 使用**编译时**和**运行时**两种 Feature Flag。

### 编译时 Feature Flag（Bun Bundle）

```
  // 代码里这样写:
  import { feature } from 'bun:bundle'

  if (feature('VOICE_MODE')) {
    // 语音相关代码
    require('./voice/index.js')
  }

  构建时发生的事:
  ┌─────────────────────────────────────────┐
  │ 如果 VOICE_MODE = true:                  │
  │   代码正常保留                            │
  │                                         │
  │ 如果 VOICE_MODE = false:                 │
  │   整个 if 块被完全删除！                   │
  │   最终产物里没有任何语音代码               │
  │   → 打包体积更小                          │
  │   → 不可能被用户意外触发                   │
  └─────────────────────────────────────────┘

  已知的编译时 Flag:
  ┌────────────────────┬────────────────────────┐
  │ BRIDGE_MODE        │ IDE 集成功能            │
  │ VOICE_MODE         │ 语音输入                │
  │ PROACTIVE          │ 自主代理模式            │
  │ KAIROS             │ 助理模式                │
  │ AGENT_TRIGGERS     │ Cron 定时调度           │
  │ COORDINATOR_MODE   │ 多代理协调              │
  │ WORKFLOW_SCRIPTS   │ 工作流脚本              │
  │ MONITOR_TOOL       │ 监控工具                │
  │ TEAMMEM            │ 团队功能                │
  └────────────────────┴────────────────────────┘
```

### 运行时 Feature Flag（GrowthBook）

```
  // 代码里这样用:
  getFeatureValue_CACHED_MAY_BE_STALE('tengu_xxx', false)

  运行时发生的事:
  ┌────────────────────────────────────────────┐
  │ 1. 检查本地缓存                             │
  │    → 有缓存？用缓存值 (可能有点过期)         │
  │    → 没缓存？去服务器拿                      │
  │                                            │
  │ 2. 后台定期刷新                             │
  │    → 从 GrowthBook 服务器获取最新值          │
  │    → 更新本地缓存                           │
  │                                            │
  │ 用途:                                       │
  │ • A/B 测试 (50% 用户开启新功能)              │
  │ • 渐进式发布 (先给 1% 用户，逐步到 100%)     │
  │ • 紧急关闭 (发现 bug 时远程关掉功能)          │
  └────────────────────────────────────────────┘

  两种缓存级别:
  ┌──────────────────────────────────────────────┐
  │ CACHED_MAY_BE_STALE (快但可能过期)            │
  │ → 适合性能敏感的路径                          │
  │ → "这个功能大概率是开着的就行"                 │
  │                                              │
  │ CACHED_OR_BLOCKING (准确但可能阻塞)           │
  │ → 适合安全相关的检查                          │
  │ → "这个功能必须确认是开着的再执行"              │
  └──────────────────────────────────────────────┘
```

## 专题 7：CLAUDE.md 加载机制

### 这是什么？

CLAUDE.md 是项目的"规则手册"——告诉 Claude 这个项目的规则、约定、注意事项。系统会从多个位置收集 CLAUDE.md 文件，合并后注入到系统提示词中。

### 加载顺序（从低到高优先级）

```
  ┌──────────────────── CLAUDE.md 发现之旅 ────────────────────┐
  │                                                            │
  │  ① 系统级 (最低优先级)                                      │
  │  /etc/claude-code/CLAUDE.md                                │
  │  → 系统管理员设置的全局策略                                  │
  │                                                            │
  │  ② 用户级                                                  │
  │  ~/.claude/CLAUDE.md                                       │
  │  → 你的个人全局偏好                                          │
  │                                                            │
  │  ③ 项目级 (向上遍历目录树)                                   │
  │                                                            │
  │  假设你在 /home/user/project/src/utils/ 工作:              │
  │                                                            │
  │  /home/user/project/src/utils/CLAUDE.md    ← 最近         │
  │  /home/user/project/src/CLAUDE.md                          │
  │  /home/user/project/CLAUDE.md              ← 项目根       │
  │  /home/user/project/.claude/CLAUDE.md                      │
  │  /home/user/project/.claude/rules/*.md     ← rules 目录   │
  │  /home/user/CLAUDE.md                      ← 继续向上     │
  │  /home/CLAUDE.md                                           │
  │  /CLAUDE.md                                ← 到根目录     │
  │                                                            │
  │  ④ 本地独有 (最高优先级，不提交 Git)                         │
  │  /home/user/project/CLAUDE.local.md                        │
  │  → 你本地独有的规则，不分享给团队                             │
  │                                                            │
  └────────────────────────────────────────────────────────────┘
```

### @include 指令——在 CLAUDE.md 中引用其他文件

```markdown
  # 项目规则 (CLAUDE.md)

  请遵循以下编码规范:
  @./docs/coding-standards.md

  API 设计指南:
  @./docs/api-guidelines.md

  我的个人偏好:
  @~/.claude/personal-rules.md
```

```
  @include 规则:
  ┌──────────────────────────────────────────┐
  │ ✅ @./relative/path   (相对路径)         │
  │ ✅ @~/home/path       (用户目录)         │
  │ ✅ @/absolute/path    (绝对路径)         │
  │                                          │
  │ ❌ 不支持在代码块里的 @                   │
  │ ❌ 循环引用自动检测并跳过                  │
  │ ❌ 不存在的文件静默忽略                    │
  └──────────────────────────────────────────┘
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/utils/claudemd.ts` | CLAUDE.md 发现、加载、@include 处理 (1600+ 行) |

## 专题 8：用户钩子 (Hooks)

### 这是什么？

钩子（Hooks）让你可以在 Claude Code 的特定时机自动运行脚本——比如每次工具执行后运行 lint，或者每次会话开始时加载环境变量。

### 支持的钩子事件

```
┌──────────── 18 种钩子事件 ────────────┐
│                                       │
│  会话生命周期:                          │
│  ├── SessionStart     会话开始时       │
│  ├── Stop             用户停止时       │
│  └── StopFailure      停止失败时       │
│                                       │
│  工具相关:                             │
│  ├── PreToolUse       工具执行前       │
│  ├── PostToolUse      工具执行后       │
│  ├── PostToolUseFailure 工具失败后     │
│  └── PermissionDenied 权限被拒绝后     │
│                                       │
│  用户输入:                             │
│  └── UserPromptSubmit 用户提交消息时   │
│                                       │
│  上下文压缩:                           │
│  ├── PreCompact       压缩前          │
│  ├── PostCompact      压缩后          │
│  └── RecoverFromCompact 压缩失败恢复  │
│                                       │
│  子代理:                               │
│  ├── SubagentStart    子代理启动时     │
│  └── SubagentStop     子代理停止时     │
│                                       │
│  通知:                                 │
│  └── Notification     通知触发时       │
│                                       │
└───────────────────────────────────────┘
```

### 怎么配置钩子？

在 `settings.json` 中：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "BashTool",
        "command": "node scripts/post-bash-check.js"
      }
    ],
    "SessionStart": [
      {
        "command": "source ~/.env.local"
      }
    ]
  }
}
```

### 钩子的输入输出协议

```
  ┌──────── 钩子怎么和 Claude Code 通信？──────────┐
  │                                                │
  │  输入: JSON 通过 stdin 传递                     │
  │  {                                             │
  │    "tool_name": "BashTool",                    │
  │    "tool_input": { "command": "npm test" },    │
  │    "response": "Tests passed"                  │
  │  }                                             │
  │                                                │
  │  输出: 通过退出码和 stdout/stderr               │
  │                                                │
  │  退出码 0:  → 成功，stdout 显示给用户/模型      │
  │  退出码 2:  → 错误，stderr 发送给模型           │
  │  其他退出码: → 警告，stderr 仅显示给用户        │
  │                                                │
  └────────────────────────────────────────────────┘
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/utils/hooks/` | 钩子管理和执行 |
| `src/schemas/hooks.ts` | 钩子事件类型和匹配器定义 |

## 专题 10：认证系统 (OAuth)

### 认证方式优先级

```
  ┌──────── API Key / Token 查找顺序 ────────┐
  │                                          │
  │ ① CLAUDE_CODE_OAUTH_TOKEN 环境变量       │
  │ ② OAuth from file descriptor             │
  │ ③ macOS Keychain / 系统密钥存储          │
  │ ④ apiKeyHelper 配置脚本                  │

  │                                          │
  └──────────────────────────────────────────┘
```

### OAuth 登录流程

```
  用户输入: /login
       │
       ▼
  ┌─────────────────────────────────┐
  │ 1. 生成安全随机数                │
  │    code_verifier (128 字节)      │
  │    code_challenge (SHA256 哈希)  │
  │    state (防 CSRF)              │
  └──────────┬──────────────────────┘
             │
             ▼
  ┌─────────────────────────────────┐
  │ 2. 打开浏览器                    │
  │    跳转到 Anthropic 登录页面     │
  │    用户在浏览器中登录+授权        │
  └──────────┬──────────────────────┘
             │
             ▼
  ┌─────────────────────────────────┐
  │ 3. 回调接收                      │
  │    本地 HTTP 服务器监听回调       │
  │    (或用户手动粘贴 auth code)    │
  └──────────┬──────────────────────┘
             │
             ▼
  ┌─────────────────────────────────┐
  │ 4. Token 交换                    │
  │    auth code → access token     │
  │    存入系统密钥存储              │
  └──────────┬──────────────────────┘
             │
             ▼
  ┌─────────────────────────────────┐
  │ 5. 获取用户信息                  │
  │    订阅类型、速率限制等          │
  └─────────────────────────────────┘
```

## 专题 11：Git Worktree 隔离

### 这是什么？

Git Worktree 让你在不离开当前代码的情况下，创建一个"平行宇宙"来做实验。Claude Code 深度集成了这个功能。

### 工作流程

```
  你: "在 worktree 里试试新的实现"
       │
       ▼
  ┌─────────────────────────────────┐
  │ EnterWorktreeTool                │
  │                                 │
  │ 1. 验证名称 (合法字符，≤64字符)   │
  │ 2. 找到 Git 主仓库根目录         │
  │ 3. 创建新分支 + worktree         │
  │ 4. 符号链接大型目录              │
  │    (如 node_modules)            │
  │ 5. 切换工作目录到 worktree       │
  │ 6. 清空缓存，重新计算上下文       │
  └─────────────────────────────────┘
       │
       ▼
  在 worktree 里正常工作...
  (文件修改不影响主仓库)
       │
       ▼
  ┌─────────────────────────────────┐
  │ ExitWorktreeTool                 │
  │                                 │
  │ action: "keep" → 保留 worktree  │
  │ action: "remove" → 删除 worktree│
  │                                 │
  │ 安全检查:                        │
  │ 如果有未提交的更改且要删除       │
  │ → 拒绝！要求先确认              │
  └─────────────────────────────────┘
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/tools/EnterWorktreeTool/` | 创建 worktree |
| `src/tools/ExitWorktreeTool/` | 退出/删除 worktree |
| `src/utils/worktree.ts` | Worktree 管理逻辑 (400+ 行) |

## 专题 13：语音模式 (Voice)

### 这是什么？

语音模式让你通过说话而不是打字来与 Claude 交互。按住键说话，松开后自动转文字。

### 录音架构

```
  ┌─────── 多平台录音方案 ────────┐
  │                               │
  │  macOS/Windows:               │
  │  audio-capture-napi           │
  │  → 原生音频 API               │
  │  → CoreAudio (macOS)          │
  │  → 16kHz 单声道               │
  │  → 首次加载约 1-8 秒           │
  │                               │
  │  Linux (有桌面):               │
  │  PulseAudio / PipeWire        │
  │                               │
  │  Linux (无桌面):               │
  │  ┌───── 降级方案 ─────┐       │
  │  │ 优先: SoX (rec)    │       │
  │  │ 备选: ALSA (arecord)│      │
  │  │ 2 秒静音自动停止    │       │
  │  └────────────────────┘       │
  │                               │
  └───────────────────────────────┘
```

### 语音转文字流程

```
  🎤 按住按键 → 开始录音
       │
       ▼
  📡 流式发送到语音识别 API
  (实时返回中间结果)
       │
       ▼
  🔤 松开按键 → 最终识别结果
       │
       ▼
  📝 文字填入输入框
       │
       ▼
  ⏎ 自动或手动提交
```
